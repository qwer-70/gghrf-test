# Этап 01. Доменные модели

## Что такое доменная модель

Доменная модель описывает важные объекты предметной области. Для магазина одежды такими объектами могут быть товар, категория, размер, остаток товара по размеру, покупатель, адрес, заказ.

Модель не обязана знать, где она хранится и как ее покажут пользователю. Ее задача - хранить состояние и защищать его от некорректных значений.

## Пример из другой области

Представим доменную модель студента. Она не связана с магазином, но хорошо показывает идею: объект хранит данные и проверяет, что они не противоречат правилам.

```python
class Student:
    def __init__(self, student_id, full_name, email, age):
        if student_id <= 0:
            raise ValueError("Идентификатор должен быть положительным")

        if full_name == "":
            raise ValueError("Имя студента не может быть пустым")

        if "@" not in email:
            raise ValueError("Некорректный email")

        if age < 16:
            raise ValueError("Возраст студента не может быть меньше 16")

        self.student_id = student_id
        self.full_name = full_name
        self.email = email
        self.age = age
```

Такая модель похожа на строку будущей таблицы, но это все еще обычный Python-объект. Она не знает, как подключаться к базе данных, в какую таблицу попадет и каким SQL-запросом ее сохранят.

Добавление объекта в PostgreSQL будет задачей репозитория, а не самой модели.

## Как использовать модели дальше

В большом проекте модели не нужно копировать из этапа в этап.

Создайте модели один раз в файле первого этапа:

```text
01_domain_models/tasks.py
```

Дальше другие этапы будут импортировать их. Из-за цифр в названиях папок обычный `from 01_domain_models...` не подойдет, поэтому можно использовать импорт по строковому имени модуля:

```python
from importlib import import_module

domain_models = import_module("17_clothing_store_project.01_domain_models.tasks")
Product = domain_models.Product
```

Так репозитории, сервисы и консольное приложение работают с одними и теми же классами. Если поле модели изменится, его нужно будет поправить в одном месте, а не искать копии по разным файлам.

## Примеры обязанностей модели

Модель может:

- хранить данные;
- проверять значения при создании объекта;
- менять свое состояние через методы;
- возвращать вычисляемые значения;
- запрещать недопустимые операции.

Модель не должна:

- читать данные через `input()`;
- напрямую печатать меню;
- сохранять себя в файл или базу данных;
- искать другие объекты в общем списке.

## Инварианты

Инвариант - это правило, которое должно оставаться верным для объекта.

Примеры:

- цена товара не может быть отрицательной;
- остаток товара конкретного размера не может быть меньше нуля;
- размер должен быть из разрешенного набора;
- email покупателя должен быть похож на email.

Если объект нарушает такие правила, лучше сразу поднять ошибку, чем хранить неправильное состояние.

## Полезные приемы

Для защиты данных можно использовать:

- проверку в `__init__`;
- методы изменения состояния;
- свойства `@property`;
- отдельные методы проверки.

На этом этапе не нужно думать о базе данных, меню и полном оформлении заказа. Сначала нужно получить надежные маленькие объекты.
