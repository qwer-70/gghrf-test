# Большое задание. Онлайн-магазин одежды

## Идея проекта

Нужно постепенно разработать консольную версию онлайн-магазина одежды на Python.

Проект строится по архитектурным слоям:

- доменные модели;
- репозитории;
- сервисы бизнес-логики;
- корзина;
- оформление заказов;
- пользователи и скидки;
- хранение данных в PostgreSQL;
- итоговое консольное приложение.

ORM и веб-фреймворки использовать нельзя. SQL-запросы нужно писать самостоятельно.

Разрешается использовать только драйвер для подключения к PostgreSQL, например `psycopg` или `psycopg2`. Остальные сторонние библиотеки в учебном проекте не нужны.

## Предметная область

Магазин продает одежду: футболки, худи, джинсы, платья, куртки и другие товары.

У товара есть название, категория, цена, цвет и признак активности. Остатки нужно хранить отдельно по размерам: например, у одной футболки может быть 3 штуки размера S и 8 штук размера M.

Покупатель может смотреть каталог, искать товары, добавлять позиции нужного размера в корзину, менять количество, применять скидку и оформлять заказ.

## Архитектура проекта

В проекте используется слоистая архитектура:

- `01_domain_models/tasks.py` - доменные модели без SQL и `input()`;
- `02_postgresql_storage/tasks.py` - подключение к БД;
- `03_repositories/tasks.py` - работа с PostgreSQL через SQL-запросы;
- `04_catalog_service/tasks.py` - бизнес-логика каталога;
- `05_cart/tasks.py` - корзина;
- `06_orders/tasks.py` - оформление заказов;
- `07_users_discounts/tasks.py` - адреса, промокоды и скидки;
- `08_console_app/tasks.py` - ввод, вывод, главное меню и сборка приложения;
- `02_postgresql_storage/schema.sql` - структура таблиц.

Это не классический MVC. Ближе всего проект устроен как `Domain Models + Repositories + Services + Console Interface`.

## Общие требования

1. Каждый этап выполняется в отдельной папке.
2. Решение каждого этапа должно запускаться отдельно.
3. Код должен быть разделен по смыслу, а не храниться целиком в одном большом классе.
4. Внутри программы данные должны передаваться через объекты моделей, а постоянное хранение должно быть в PostgreSQL.
5. Для хранения данных используется реальная база данных PostgreSQL.
6. Нельзя использовать ORM, Django, Flask, FastAPI, SQLAlchemy ORM, Pydantic и другие инструменты, которые скрывают работу с SQL.
7. Ошибочные ситуации должны обрабатываться явно: неверный размер, отрицательная цена, нехватка товара на складе, пустая корзина.
8. Методы должны возвращать результат, а не только печатать текст.
9. Печать в консоль лучше держать ближе к интерфейсному слою.
10. Параметры подключения к БД не нужно хранить прямо в коде: используйте переменные окружения или отдельный локальный файл, который не попадает в репозиторий.

Краткая инструкция по подключению к PostgreSQL находится в файле `DATABASE.md`.

Важно: этапы идут как единый путь разработки. В каждом следующем этапе нужно импортировать код из предыдущих этапов и дорабатывать проект дальше. Финальная версия собирается в этапе 08.

Код проекта не нужно дублировать. Например, модель товара создается один раз в `01_domain_models/tasks.py`, а репозиторий использует импорт из этого файла.

Так как папки курса начинаются с цифр, обычная запись вида `from 01_domain_models.tasks import Product` не подходит. Для учебного проекта можно импортировать предыдущий этап через `importlib.import_module()` по строковому имени модуля.

## Этапы проекта

### 01. Доменные модели

Создать основные сущности магазина: товар, категория, размер или остаток товара по размеру, покупатель. На этом этапе важно научиться описывать предметную область через классы и проверять корректность данных.

### 02. PostgreSQL: база, схема и подключение

Создать реальную PostgreSQL-базу данных, подготовить `schema.sql`, описать первые таблицы, связи и ограничения, настроить подключение из Python.

### 03. Репозитории

Добавить слой PostgreSQL-репозиториев поверх готовой схемы. Репозиторий должен отвечать за добавление, получение, обновление, удаление и поиск объектов через SQL-запросы, но не должен принимать бизнес-решения.

### 04. Сервис каталога

Добавить бизнес-логику каталога: поиск товаров, фильтрацию, сортировку, проверку доступности конкретного размера, скрытие неактивных товаров.

### 05. Корзина

Реализовать корзину покупателя: добавление товаров, изменение количества, удаление позиции, подсчет суммы и очистка.

### 06. Заказы

Реализовать оформление заказа из корзины. Заказ должен фиксировать позиции, размеры, цены на момент покупки, итоговую сумму, данные покупателя и статус.

### 07. Пользователи и скидки

Добавить учет покупателей, адресов доставки и промокодов. Скидки должны применяться через отдельный слой логики.

### 08. Итоговое приложение

Собрать готовое консольное приложение, где пользователь может просмотреть каталог, найти товар, собрать корзину и оформить заказ.

## Минимальный итоговый сценарий

Финальная программа должна позволять выполнить такой сценарий:

1. Загрузить стартовые товары.
2. Показать каталог активных товаров.
3. Найти товар по названию или категории.
4. Добавить товар нужного размера в корзину.
5. Изменить количество товара в корзине.
6. Посмотреть итоговую сумму.
7. Применить промокод, если он существует.
8. Оформить заказ.
9. Уменьшить остатки товаров выбранных размеров на складе.
10. Сохранить заказ в PostgreSQL.

## Критерии готовности

Проект считается выполненным, если:

- каждый этап запускается без синтаксических ошибок;
- классы не смешивают разные обязанности;
- репозитории не печатают данные и не спрашивают ввод;
- репозитории работают с PostgreSQL через SQL-запросы;
- остатки товаров хранятся и проверяются по конкретному размеру;
- сервисы не зависят от `input()`;
- интерфейсный код можно заменить без переписывания моделей;
- ошибки пользователя не ломают программу;
- финальное приложение демонстрирует полный путь от каталога до заказа.
