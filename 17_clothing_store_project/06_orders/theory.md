# Этап 06. Заказы

## Что такое заказ

Заказ фиксирует покупку. В отличие от корзины, заказ должен хранить состояние на момент оформления: выбранные товары, размеры, количество, цену за единицу, сумму и покупателя.

После создания заказа корзину можно очистить, а остатки товаров выбранных размеров на складе нужно уменьшить.

Позиция заказа должна быть снимком покупки. Если позже у товара изменится название или цена, уже созданный заказ должен остаться понятным.

## Статус заказа

У заказа может быть статус:

- создан;
- оплачен;
- передан в доставку;
- выполнен;
- отменен.

Для учебного проекта достаточно нескольких статусов. Важно, чтобы статус менялся через метод, а не случайным присваиванием из любого места.

## Сервис заказов

Оформление заказа - это бизнес-операция. Она затрагивает корзину, товары, покупателя и остатки на складе.

Поэтому лучше сделать отдельный сервис заказов. Он будет:

- проверять корзину;
- проверять остатки;
- создавать заказ;
- списывать товары со склада по конкретным размерам;
- сохранять заказ в репозитории.

## Таблицы заказов

Когда в проекте появляются заказы, SQL-схему нужно расширить.

Обычно нужны две таблицы:

- `orders` - общая информация о заказе;
- `order_items` - позиции заказа.

Позиция заказа должна ссылаться на заказ и товар через внешние ключи. При этом название товара, размер и цена на момент покупки могут храниться прямо в позиции заказа как снимок данных.

## Транзакция оформления

Оформление заказа должно быть транзакцией.

В одной операции нужно:

- создать заказ;
- создать позиции заказа;
- уменьшить остатки выбранных размеров;
- очистить корзину после успешного оформления.

Если один шаг завершился ошибкой, изменения в БД нужно откатить.

## Ошибочные ситуации

Нужно продумать, что делать, если:

- корзина пустая;
- покупатель не найден;
- товара стало меньше, чем было в момент добавления в корзину;
- выбранный размер больше недоступен.

Чем раньше эти ситуации описаны явно, тем проще сопровождать проект.

## Пример из другой области

Представим запись клиента на занятие в спортивный зал. Пока клиент выбирает занятие, это еще не запись. Запись появляется только после подтверждения.

```python
class TrainingBooking:
    def __init__(self, booking_id, client_name, training_title):
        self.booking_id = booking_id
        self.client_name = client_name
        self.training_title = training_title
        self.status = "created"

    def cancel(self):
        if self.status == "completed":
            raise ValueError("Нельзя отменить завершенную запись")

        self.status = "cancelled"


class BookingService:
    def create_booking(self, client, training):
        if not training.has_free_places():
            raise ValueError("На занятии нет свободных мест")

        training.reserve_place()
        return TrainingBooking(1, client.name, training.title)
```

Сервис выполняет бизнес-операцию: проверяет возможность записи, меняет состояние занятия и создает объект записи.
