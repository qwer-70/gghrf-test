# Этап 03. Репозитории

## Зачем нужен репозиторий

Репозиторий - это слой, который отвечает за сохранение и получение объектов.

В этом проекте репозитории работают с реальной базой данных PostgreSQL, подготовленной на предыдущем этапе. Это значит, что данные не исчезают после завершения программы и могут быть прочитаны при следующем запуске.

Репозиторий помогает отделить вопрос "где лежат данные" от вопроса "что с ними делать".

## Репозитории без ORM

В проекте нельзя использовать ORM: Django ORM, SQLAlchemy ORM, Peewee и похожие инструменты.

Разрешается использовать только драйвер для подключения к PostgreSQL, например `psycopg` или `psycopg2`.

SQL-запросы нужно писать самостоятельно:

- `INSERT` для добавления;
- `SELECT` для чтения;
- `UPDATE` для изменения;
- `DELETE` для удаления.

## Что должен делать репозиторий

Обычно репозиторий умеет:

- добавить объект в таблицу;
- получить объект по идентификатору;
- вернуть список объектов;
- обновить объект;
- удалить объект;
- проверить существование объекта.

Репозиторий не должен решать, можно ли купить товар, какую скидку дать покупателю и как оформить заказ. Это задачи сервисов.

## Импорт моделей

Репозитории не должны заново объявлять модели.

Модель создается один раз в `01_domain_models/tasks.py`, а репозиторий импортирует ее:

```python
from importlib import import_module

domain_models = import_module("17_clothing_store_project.01_domain_models.tasks")
Product = domain_models.Product
```

Так в проекте остается один источник правды для структуры объекта.

## С какими таблицами работаем

На этом этапе репозитории работают с базовой схемой каталога:

- `categories` - категории одежды;
- `products` - основные данные товара;
- `product_stocks` - остатки конкретного товара по конкретному размеру;
- `customers` - покупатели.

Остатки лучше не хранить одним числом в таблице товаров. Для магазина одежды важно знать количество отдельно по каждому размеру.

## Соединение с базой данных

Репозиторий может получать готовое соединение с базой данных через `__init__`.

Так проще контролировать, кто подключается к БД, и не создавать новое соединение внутри каждого метода.

```python
class SomeRepository:
    def __init__(self, connection):
        self.connection = connection
```

Функция подключения уже должна быть подготовлена на этапе 02. Репозитории не должны сами хранить пароль, читать `input()` или создавать скрытое новое подключение внутри каждого метода.

## Использование подключения

В основном файле приложения можно создать подключение один раз, передать его в репозитории, а в конце закрыть.

```python
connection = get_connection()

try:
    book_repository = BookRepository(connection)
    # дальше приложение работает с репозиторием
finally:
    connection.close()
```

Для учебного проекта этого достаточно. В больших приложениях управление соединениями обычно сложнее, но идея остается той же: репозитории не должны сами хранить пароль и создавать подключение тайно внутри каждого метода.

## Пример из другой области

Представим приложение для библиотеки. Книга - это модель, а репозиторий книг сохраняет книги в PostgreSQL.

```python
class Book:
    def __init__(self, book_id, title, author):
        self.book_id = book_id
        self.title = title
        self.author = author


class BookRepository:
    def __init__(self, connection):
        self.connection = connection

    def add(self, book):
        query = """
            INSERT INTO books (id, title, author)
            VALUES (%s, %s, %s)
        """

        with self.connection.cursor() as cursor:
            cursor.execute(query, (book.book_id, book.title, book.author))

        self.connection.commit()

    def get_by_id(self, book_id):
        query = """
            SELECT id, title, author
            FROM books
            WHERE id = %s
        """

        with self.connection.cursor() as cursor:
            cursor.execute(query, (book_id,))
            row = cursor.fetchone()

        if row is None:
            return None

        return Book(row[0], row[1], row[2])
```

Это не решение задания про магазин. Пример показывает сам принцип: репозиторий выполняет SQL-запросы и превращает строки из таблицы обратно в объекты.

## Параметры запроса

Значения нельзя вставлять в SQL через f-строки.

Плохо:

```python
query = f"SELECT * FROM books WHERE id = {book_id}"
```

Лучше передавать значения отдельно:

```python
cursor.execute("SELECT * FROM books WHERE id = %s", (book_id,))
```

Так код защищается от ошибок с кавычками и от SQL-инъекций.

## Транзакции

Изменения в PostgreSQL нужно фиксировать через `commit()`.

Если во время операции произошла ошибка, можно выполнить `rollback()`, чтобы отменить незавершенные изменения.

Транзакции особенно важны при оформлении заказа: нужно одновременно создать заказ, создать позиции заказа и уменьшить остатки товаров по выбранным размерам.

## Ошибки

Если объект не найден, можно:

- вернуть `None`;
- поднять исключение;
- сделать отдельный метод проверки.

Важно выбрать один понятный подход и использовать его одинаково.
